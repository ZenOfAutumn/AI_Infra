# 动态图与静态图 (Dynamic vs Static Graphs)

在深度学习框架中，理解计算图的两种执行模式至关重要。计算图是深度学习框架用来表示数学运算和数据流的数据结构。

## 1. 原理比较

*   **动态图 (Dynamic Graph / Eager Execution)**:
    *   **原理**: "Define-by-run"（边运行边定义）。计算图在代码执行时动态构建。每执行一行 PyTorch 代码，底层就会立即分配内存、调用算子并计算结果，同时记录下这一步操作以便后续反向传播（Autograd）。
    *   **特点**: 就像普通的 Python 程序一样，可以使用原生的 Python 控制流（`if/else`, `for`, `while`）。

*   **静态图 (Static Graph / Graph Execution)**:
    *   **原理**: "Define-and-run"（先定义后运行）。分为两个阶段：首先，框架会解析代码或追踪执行路径，构建出一个完整的、独立于 Python 的计算图（中间表示 IR）；然后，将数据输入这个编译好的图中进行高效执行。
    *   **特点**: 计算图一旦构建完成就固定了。控制流需要被特殊处理（如转换为图中的条件节点 `prim::If`）。

## 2. 优缺点比较

| 特性 | 动态图 (Eager Mode) | 静态图 (Graph Mode) |
| :--- | :--- | :--- |
| **易用性与调试** | **极佳**。可以使用 Python 的 `print`、`pdb` 随时查看张量的值，代码直观。 | **较差**。图构建阶段没有实际数据，运行时报错难以定位到具体的 Python 代码行。 |
| **控制流** | **原生支持**。直接使用 Python 的 `if/for`。 | **受限**。需要框架支持（如 TorchScript 的 `jit.script`）才能将控制流编译进图中。 |
| **执行性能** | **较低**。每次运算都有 Python 解释器的开销（Python Overhead），无法进行全局的图优化。 | **极高**。脱离 Python 解释器（C++ 运行），支持**算子融合 (Kernel Fusion)**、死代码消除、内存预分配等全局优化。 |
| **部署 (Deployment)** | **困难**。强依赖 Python 环境，难以直接部署到 C++、移动端或边缘设备。 | **容易**。可以序列化为独立的文件（如 `.pt`, `.onnx`），在无 Python 环境的推理引擎中加载运行。 |

## 3. PyTorch 实现的区别

PyTorch 默认是**动态图**框架，但提供了多种机制将其转换为**静态图**以提升性能和部署能力：

1.  **`torch.jit.trace` (追踪法)**:
    *   **用法**: 提供一个示例输入，PyTorch 运行一遍模型，记录下所有执行过的算子。
    *   **局限**: **无法捕获控制流**。如果代码中有 `if flag > 0:`，它只会记录示例输入走过的那一个分支，另一个分支会被永久丢弃。
2.  **`torch.jit.script` (脚本法)**:
    *   **用法**: 解析 Python 代码的抽象语法树 (AST)，将其编译为 TorchScript IR。
    *   **优势**: **可以保留控制流**。它会将 `if/else` 转换为图中的 `prim::If` 节点。
3.  **`torch.compile` (PyTorch 2.0+)**:
    *   **用法**: 最新的 JIT 编译方案。通过 `model = torch.compile(model)`，底层使用 TorchDynamo 动态捕获计算图，并使用 OpenAI Triton 生成高度优化的 GPU 机器码。这是目前 LLM 训练和推理加速的主流方案。
